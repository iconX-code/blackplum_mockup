## 1. 제품 개요

### 1-1. definition

- 프로젝트명 (내부용): INBOXX — 아이콘엑스 내부 전용 프로젝트명
- 제품 브랜드: blackplum (블랙플럼)
- 포지셔닝: "메시지를 모아주는 툴"이 아니라 "처리해야 할 일을 보여주는 업무 큐"
- 핵심 차별점: 비즈니스 / 운영 / 팬소통 자동 분리 + 인스타그램 DM 자동화
- 비즈니스 목표: 크리에이터/매니저의 "운영용 필수 툴" → 향후 팀 협업툴·브랜드 협업툴 확장 기반

### 1-2. target user
- instagram & youtube 중심 중형급 크리에이터 (공구/시딩/광고/협찬 문의가 실제 유입되는 계정)
- 핵심 문제
	- ① 중요 메시지 매몰
	- ② 반복 대응 공수
	- ③ 메시지→업무 전환 불가

### 1-3. target environment
- open spec 기준, 반응형 web으로 구현하여 다양한 기기 커버 (런칭 후 시장 반응에 따라 hybrid app으로 패키징 예정)
- 잠재고객 인터뷰 결과에 따른 서비스이용환경 예상 비중: 모바일 > PC/맥북 > 태블릿
- UI 품질 우선순위: 모바일 > PC 웹 > 태블릿. 모바일 화면의 완성도를 가장 높게 유지한다
- 디바이스별 실질 기능 차이는 없으나, 모바일에서는 화면 단순화를 위해 일부 minor UI 요소를 숨기거나 생략할 수 있으며, 이로 인한 부차적 기능 축소는 감내한다 (상세: §8-4)

## 3. 채널(워크스페이스) 설계

### 3-1. 기본 개념

- 채널 = 워크스페이스 = Company: 1셀럽 = 1채널 기준
- 채널명 (채널닉네임): 'display name', 중복가능, 채널생성 시 입력받음
- 채널 ID = slug: globally unique, 채널별 url handle 역할, 채널생성 시 1)채널명이 영문이면 일단 자동 채우기(단, 중복이면 임의 자동 suffix 붙이기) 2)한글이면 빈칸 3)유저가 채워넣거나 수정할 수 있도록 제공
- 채널 생성 자격: 누구나 가능, 생성 시 본인이 생성한 채널에 대한 어드민 권한 자동부여
- 데이터 정책: 회사:계정 = N:M (운영 유연성), MVP에서는 N:M 개념 프론트엔드 미노출
- 채널 삭제: 어드민 관점에서는 '채널 삭제' 메뉴만 존재함. trigger 시, 실제로는 soft delete -> 30일 staging -> hard delete 방식으로 동작.
	- 삭제 trigger 시, 기존에 남아있던 멤버들에게 안내 메일 자동발송
- 채널 탈퇴: 계정탈퇴를 하지 않고 채널에서만 나올 수 있다. (설정 혹은 내정보 메뉴에서 제공)
	- 동일멤버가 채널탈퇴 후 재참여 시 기존 활동이력과 연결한다.

### 3-2. 계정 Hierarchy

| 역할  | 설명                        |
| --- | ------------------------- |
| 어드민 | 채널 생성자가 자동 부여. 채널당 1명만 존재 |
| 멤버  | 초대받아 참여한 참여자              |
- 한 유저는 채널에 따라 서로 다른 역할일 수 있음. 즉 채널1에서는 어드민, 채널2에서는 멤버 역할일 수 있음
- 어드민/멤버 간 작업가능범위에는 대부분 차이가 없으나, 아래 세가지 작업은 어드민만 trigger 가능
	- 채널 삭제
	- 어드민 권한 이양
	- 추방하기
- `[MOCKUP]` 계정탈퇴, 채널탈퇴, 멤버이탈 처리 등 상세 플로우는 목업에서 미구현 (MVP spec §2-4, §3-2 참조)

### 5-2. Flat Tag 기반 분류체계 - WIP

기본 원칙: 분류 체계는 flat(단층) tag 체계를 기반으로 한다.

| 개념              | 정의                                                                                     |
| --------------- | -------------------------------------------------------------------------------------- |
| 태그 (Tag)        | 메시지 스레드에 부착되는 실질적 분류 단위. 한 스레드에 여러 개의 태그가 부착될 수 있음. 아무런 태그도 붙지 않은 스레드는 '직접확인' 에서 확인 가능 |
| 카테고리 (Category) | 실질적 분류 체계가 아닌 view filter에 해당. 특정 태그가 붙은 메시지를 모아서 보여주는 역할                              |
- 'AI pick','전체함', '저장됨' 에서는 모든 tag가 가용하다.(노출된다.) `[MOCKUP]` 전체함/저장됨 미구현이므로 AI pick에서만 전체 tag 가용
- 다른 카테고리에서는 해당 카테고리에 해당하는 tag만 가용하다.(노출된다.)

태그 목록 (preset):

| 카테고리 <br>(view filter) | 포함 태그                                                                                     |
| ---------------------- | ----------------------------------------------------------------------------------------- |
| 비즈니스                   | 공구 제안 · 시딩 제안 · 광고/협찬 제안 · 행사/오프라인 제안 · 기타 제안                                             |
| 운영/CS                  | 구매 문의 · 배송 문의 · CS/환불 · 제품 질문 · 링크 요청                                                     |
| 소통                     | 셀럽 · VIP팬 · 일반팬 · 브랜드 · 고민상담 · 콘텐츠 요청<br>(셀럽/VIP팬/일반팬 태그는 side panel에서 유저가 sender별 부착 가능) |
| 정크                     | (스팸/유해성 관련 태그)                                                                            |

- 한 스레드에 태그가 여러 개 부착될 수 있으므로, 카테고리별 뱃지 합계가 실제 메시지 수를 초과할 수 있음
- 카테고리는 특정 태그 조합을 필터링하여 보여주는 뷰(view)이므로, 동일 스레드가 복수 카테고리에 노출될 수 있음
- 태그 칩의 노출 우선순위는 static하게 정하도록 한다. (WIP)
- AI pick 탭: 별도 태그 없음. 메시지 왼쪽에 원래 태그 + 오른쪽에 Context(게시물 썸네일 등) 표시
- MVP에서 태그는 preset only, non-customizable (커스텀 태그는 고도화 범위)
- 첨부파일(이미지 포함)은 분석 대상 아님. 파일명이 가용하면 분석에 포함(첨부와 본문 내용이 달라 오탐될 가능성은 감수)
- '중요' / '저장' 태그 (sender 기반 수동 우선순위)
	- sender가 중요한 사람이어서 메시지 내용과 무관하게 높은 우선순위를 부여하고 싶은 경우, 유저가 스레드에 '중요' 또는 '저장' 태그를 수동 지정 가능
	- '저장' 태그가 부착된 스레드는 '저장됨' 카테고리에서 모아보기 가능
	- 부착된 스레드는 우선순위 scoring에 가산되어 상위에 노출됨 (상세는 6-1 참조)

### 5-3. Context 관리 체계

분류 정확도 향상을 위해 메시지/스레드 단위 context를 별도로 기록·관리하며, 태그 분류는 context가 반영된 상태에서 수행한다.

#### 5-3-1. 메시지 단위 context

| 구분       | 정의                                                            |
| -------- | ------------------------------------------------------------- |
| 메시지 단독맥락 | 해당 메시지 자체의 내용만으로 추출되는 맥락                                      |
| 메시지 누적맥락 | 직전 메시지의 누적맥락 + 현재 메시지의 단독맥락을 결합한 맥락<br>(이전 맥락 위에 새로운 정보가 쌓이는 구조) |

- 메시지마다 단독맥락과 누적맥락을 함께 기록
- 같은 스레드의 2번째 이상 메시지부터는 직전 메시지의 누적맥락을 기반으로 현재 메시지의 누적맥락을 생성
- 태그 부착 기준: 메시지 누적맥락
	- 예: (시딩 논의 중) '예, 그러시죠' → 단독맥락 기준으로는 미분류되나 누적맥락 기준으로는 '시딩' 태그 부착

#### 5-3-2. 스레드 단위 context (스레드 맥락)

- 스레드 전체의 context를 별도로 관리
- 갱신 주기: 토큰 최적화를 위해 매 메시지마다 갱신하지 않고 띄엄띄엄 갱신
- 최신 메시지 누적맥락과 스레드 맥락은 많은 경우 일치할 가능성이 높으나, 동작 원리 상 차이 존재
	- 메시지 누적맥락: 오래된 history가 점차 자연스럽게 희석됨
	- 스레드 맥락: 주제 변경 시에도 이전 주제를 유실하지 않도록 prompting
- 주제 변경 판정 기준: 메시지별 태그 구성이 직전 메시지 대비 변경되면, 해당 메시지 기준으로 주제가 바뀐 것으로 보고 직전 메시지까지를 한 맥락 덩어리로 취급
- 스레드가 충분히 길어지면, 스레드 맥락은 스레드의 역사요약, 메시지 누적맥락은 현안에 대한 요약이 됨

#### 5-3-3. 예시

예1: DM case

| 구분       | 내용                                                              |
| -------- | --------------------------------------------------------------- |
| 스레드 맥락   | 본 스레드는 주제 A→B→C→B→D 순서로 논의가 진행되고 있음. A, B, C는 완료된 사안이고, 현안은 D |
| 메시지 누적맥락 | D에 대한 논의 중. 상대방이 OOO를 제안해, 셀럽이 답변할 차례                          |
| 메시지 단독맥락 | 상대방의 의견을 구함                                                     |

예2: 댓글 case

| 구분       | 내용                                                                                                            |
| -------- | ------------------------------------------------------------------------------------------------------------- |
| 스레드 맥락   | 4/10부터 시작한 yeti 텀블러 공구 게시글에 달린 댓글 스레드. 다음에는 stanley 텀블러도 진행해달라는 원댓글에 대해 stanley 텀블러 중 어떤 제품을 원하는지에 대한 대댓글이 다수 달림 |
| 메시지 누적맥락 | 230mL 제품과 475mL 제품 중 뭐가 좋은지에 대해 논의 진행 중                                                                       |
| 메시지 단독맥락 | 475mL 녹색을 원함                                                                                                  |

- 스레드 맥락 기록 기능을 활용해, 시계열 반응 모니터링 등 맥락 history 재가공 서비스 제공 가능

### 5-4. Origin 별 수동 tagging 기능 `[MOCKUP]`

	- 긴급
	- 이벤트-예정 / 이벤트-오픈 / 이벤트-종료
	- VIP (high priority) / NIP (low priority) 등
- 유저가 손수 설정한 tagging 규칙에 의해 붙은 태그는 우선순위 scoring에서 높은 weight 부여
- 설정 UI: 섹션3(side panel)에서 스레드 상세 확인 중 Origin 별 tagging 규칙을 지정할 수 있는 UI 제공 (예: Origin 정보 영역에 "이 Origin에 태그 규칙 추가" CTA)

## 6. 핵심 기능 상세

### 6-1. 우선순위 score에 따른 스레드 listing

- 각 탭(카테고리) 내 스레드 리스트는 기본적으로 시간순이 아닌 우선순위 score 기준으로 정렬
- 오픈스펙에서는 '스레드' 단위의 scoring만 고려한다 (메시지 단위 scoring은 post-launch backlog)
- scoring logic은 WIP (아래 컨셉 적용하여 수식화 예정)
- 비활성 스레드('처리 안 한 메시지'가 없는 스레드)에 대해서도 score는 유지됨. 다만 이미 모든 메시지가 '처리' 완료되었으므로 AI pick 및 다른 뷰 필터에서 노출되지 않을 뿐임
- 비즈니스 탭 우선순위 고려
	- 제안서/파일 첨부 있음
	- 금액 언급 있음
	- 공구/단독 제안
	- 브랜드명/회사명/담당자 정보가 분명함
	- 응답 시한이 명시됨
	- 재문의/후속 연락임
- 운영/CS 탭 우선순위 고려
	- 구매 의도 수준에 따라 우선순위화
	  (구매 직결 판단 기준: ① 구매/주문 의도 명시 ② 거래 변수 질문 ③ 즉시 구매 전환 가능성)
	- 높음: 구매 행동 진입/직전 막힘 (예: 배송·결제 오류, 링크 불량, 재고 문제, 환불·교환·누락)
	- 중간: 구매 의도 명확 (예: "어디서 사요?", "링크 주세요", "가격 얼마?")
	- 낮음: 구매 의도 불명확한 정보 탐색 (예: 성분·사용법·후기 질문)
- 콘텐츠 생성일(created 시점) 반영
	- 대부분의 이벤트는 시점-sensitive하게 진행되며, 이벤트 진행 기간은 수 주를 넘기지 않을 것으로 예상
	- 상기 전제 수용 시, 콘텐츠의 created 시점의 최신성과 중요도는 비례한다고 볼 수 있음
	- 오래된 콘텐츠일수록 scoring 비중을 낮게 두어, 과거 이벤트 스레드가 불필요하게 높은 우선순위를 부여받는 현상 방지
- sender 기반 수동 우선순위 가산
	- 메시지 내용과 무관하게 sender가 중요한 경우, 유저가 스레드에 '중요' 또는 '저장' 태그를 수동 지정하면 우선순위 score에 가산
	- '저장' 태그 부착 스레드는 '저장됨' 카테고리에서 모아보기 가능


### 6-2. 인스타그램 DM 자동화

블랙플럼이 연동된 Instagram 계정의 게시물에 댓글이 달리면, 사전 정의한 룰에 따라 댓글 작성자에게 DM을 자동 발송하고 동시에 해당 댓글에 자동 답글을 남기는 기능. 크리에이터가 잠들어 있는 시간에도 24시간 응답 가능하도록 만들어, 미응답으로 인한 기회손실(공구 좌표 요청, 링크 요청 등)을 줄이는 것이 목적이다.

`[MOCKUP]` 본 mockup에서는 시연 가능한 범위로 P0+P1+P2 일부를 구현한다. 발송 엔진·IG API 연동·rate limit 등 백엔드 동작은 시뮬레이션하지 않으며, 룰 작성·편집·토글·통계 표기 등 UI 시연만 구현한다. 상세 PRD는 `docs/macro-prd.md` 참조 (구현 우선순위 §10 P0+P1+P2 일부 채택).

#### 6-2-6. 검증 규칙 (룰 저장 직전)

저장(=`완료` CTA) 시점에 모든 조건이 충족되어야 ACTIVE로 활성화된다. 미충족 항목이 하나라도 있으면 해당 step에서 inline 에러를 노출하고 다음 step 이동을 차단한다 (macro-prd.md §6).

| 조건 | 필수 검증 |
| --- | --- |
| `target_mode = SPECIFIC_POSTS` | `target_post_ids` ≥ 1 |
| `trigger_mode = KEYWORD_MATCH` | `trigger_keywords` ≥ 1 |
| `reply.enabled = true` | `reply.items` ≥ 1 |
| `message.type = IMAGE_CAROUSEL` | `message.images` ≥ 1 |
| 모든 케이스 | `message.body` non-empty |
| 버튼 추가됨 | `button.text` non-empty AND `button.url` non-empty |
| `follower_branch.mode = DIFFERENT_MESSAGES` | follower_message·non_follower_message 둘 다 위 조건을 각각 만족 |

### 6-3. 빠른 답장 / 저장 답변
- 메시지 상세 페이지에서 실제 답변한 내용을 바로 저장
- 저장된 답변은 같은 카테고리 메시지에서만 노출
- UI: dropdown/자동추천chips 등 (디자인 단계에서 확정)
- `[MOCKUP]` AI 초안 생성 기능 추가: 기존에 이용한 문구 재활용뿐 아니라, 해당 스레드 맥락을 기반으로 AI가 답변 초안을 자동 생성. 유저가 초안을 편집 후 발송하거나 폐기할 수 있음

### 6-4. 게시글/콘텐츠 관제 기능 `[MOCKUP]`

### 8-4. 반응형 디자인 & 모바일 최적화

모바일, PC 웹, 태블릿 간 실질 기능 차이는 없다. 단, 모바일 화면은 가장 높은 UI 품질 기준을 적용하여 깔끔하고 직관적인 화면을 최우선으로 구성한다.

모바일 UI 설계 원칙:

- 화면 단순화 우선: 너무 많은 UI 요소로 사용자의 시선이 분산되지 않도록, 핵심적이지 않은 UI 요소는 숨기거나 생략한다. 이로 인해 PC 버전 대비 minor한 기능 축소가 발생하더라도 감내한다
- fat finger 방지: PC에서는 문제없는 작은 clickable 요소도 모바일에서는 오터치를 유발할 수 있으므로, clickable area의 최소 크기를 확보한다 (최소 44x44px 권장). 확보가 어려운 경우 핵심 기능이 아니면 해당 요소를 숨기는 것도 고려한다
- chip/dropdown/button 정리: 서비스 특성상 chip, dropdown, button이 많은 UI이므로, 모바일에서는 이들이 번잡해 보이지 않도록 정리한다. 한 화면에 노출되는 수량이 과다한 경우, 우선순위가 낮은 항목부터 접기(fold)/더보기(more) 처리하거나 생략한다
- 숨김/생략 판단 기준: 해당 UI 요소가 현재 화면의 핵심 태스크(메시지 확인, 답장, 처리) 수행에 필수적인지 여부. 필수적이지 않은 보조 정보·부가 기능은 숨김 대상
